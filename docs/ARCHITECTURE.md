## Components

### Frontend

- Mobile-first browser app (React or vanilla); chat-style turn-by-turn interaction.
- Tamil-language UX (start). English fallback for testing.
- Image capture (smartphone camera, natural lighting; plan for blur, fingers in frame).
- Text input is the default. Voice/TTS is **out of baseline scope** — add only if the field-deployment population specifically requires hands-free interaction.

### Orchestrator (DSPy Agent)

- Pattern: **ReAct + hierarchical multi-agent**.
- Routes the conversation through the 7-stage flow (see "Conversation flow" below).
- Maintains conversation state, decides when to ask clarifying questions vs. when to call tools.
- DSPy signatures: `NarrativeToSOAP`, `TriageDecision`, `PersuasionFraming`.

### MCP Server 1: Image Classifier

- **Base model:** ResNet-50 (fastest), EfficientNet-B4 (best accuracy/cost), or ViT (best generalization).
- **Training data:** curated Stanford Skin Dataset subset — most common, most dangerous, most visually distinct conditions for the chosen region (pneumonia-like, infection, fungal, allergic, parasitic).
- **Output:** top-3 predictions with confidence scores.
- **Bias control:** training set must be tone-balanced; report accuracy stratified by skin tone separately.
- **Hardware:** fine-tune on MI300X (8-16 hours expected per the brief).

### MCP Server 2: SOAP Extractor

- **Method:** DSPy `NarrativeToSOAP` signature, optimized via `BootstrapFewShot` (or `BootstrapInContext`).
- **Training data:** 50-100 hand-labeled (narrative, ground-truth SOAP) pairs from the chosen village or proxy.
- **Output fields:** chief complaint, HPI, past medical history, medications, vitals (if stated), physical exam.
- **Hard cases:** colloquialisms, negations ("no fever" vs. "I feel feverish"), irrelevant context (the neighbor's opinion, the harvest schedule).

### MCP Server 3: Village Context

- **Purpose:** convert clinical urgency into *practical* urgency by accounting for local barriers.
- **Schema** (Neo4j or JSON, derived from CareGraph):
  - Village → Clinics (distance, hours, services, lab capacity)
  - Clinics → Drugs (availability, cost in local currency)
  - Transport (bus routes, fares, schedules, seasonality)
  - Seasonal calendar (crop cycles, peak labor months)
- **Output:** structured barriers + cost-benefit framing data for the Triage Reasoner.

### MCP Server 4: Triage Reasoner (RLVR-optimized)

- **Base model:** Qwen-2.5 7B with LoRA adapter.
- **Optimization:** GRPO (Group-Relative Policy Optimization) with verifiable reward.
- **Reward function:**
  ```
  R(predicted, ground_truth) = 1.0  if exact match
                              0.5  if adjacent level (Yellow when truth is Red)
                              0.0  if off by 2+
  ```
- **Inputs:** image top-3 + SOAP fields + village context.
- **Output:** urgency level (Red/Yellow/Green) + reasoning trace + persuasion framing.
- **Training data:** 50-100 labeled cases (narrative + image + ground-truth urgency).
- **Hardware:** LoRA fine-tune on MI300X (single instance has enough HBM3 for Qwen 7B + vision model + serving).

### Output: SOAP Summary

Structured pre-visit note transmitted via SMS (or printed for hand-carry) to the clinic. Fields:

- Patient identifiers (with consent, redactable)
- Chief complaint, history, associated symptoms
- Physical findings via phone image
- Probable differential (preliminary, ranked)
- Red flags noted
- Barriers to care (informs whether the doctor's prescription is actionable)
- Plan summary (what the patient committed to)
- **Note:** "Generated by Path to Care mobile health assistant, not a licensed provider. For diagnostic and treatment decisions, please rely on clinical judgment."

## Conversation flow (7 stages — the Rajan dialogue)

1. **Image intake** — chat prompt + smartphone photo of affected area.
2. **Subjective intake** — natural-language story (typed) → structured SOAP fields via DSPy.
3. **Red-flag detection** — vision + clinical rule engine cross-check.
4. **Triage + contextualization** — Yellow-to-Red urgency with practical barriers.
5. **Persuasion** — reframes urgency as cost-benefit ("one afternoon + tablets" vs. "lose 3 days of work").
6. **Provider summary** — pre-visit SMS for the clinic doctor.
7. **Engineering reflection** — system logs what it did and didn't do (for audit).

## Serving

- **vLLM** on MI300X for Qwen 7B inference.
- **PyTorch** for vision model inference (single Triton kernel call per image).
- **MCP servers** as separate processes (FastAPI), orchestrator calls via JSON-RPC.
- **HF Space** for the public demo — likely Gradio frontend wired to a hosted backend (or a self-contained Space if the model fits).
